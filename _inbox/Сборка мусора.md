Сборка мусора в основном сводится к уничтожению короткоживущих объектов с небольшим временем жизни. Для оптимизации производительности сборщика мусора управляемая куча делится на три поколения: 0, 1 и 2. Следовательно, объекты с большим и небольшим временем жизни обрабатываются отдельно. Сборщик мусора хранит новые объекты в поколении 0. Уровень объектов, созданных на раннем этапе работы приложения и оставшихся после сборок мусора, повышается, и они сохраняются в поколении 1 и 2. Так как сжать часть управляемой кучи быстрее, чем всю кучу, эта схема позволяет сборщику мусора освобождать память в определенном поколении, а не для всей кучи при каждой сборке мусора.

- **Поколение 0**: это поколение является самым молодым и содержит кратковременные объекты. Примером короткоживущего объекта является временная переменная. Сборка мусора чаще всего выполняется в этом поколении.
    
    Вновь распределенные объекты образуют новое поколение объектов и неявно являются сборками поколения 0. Тем не менее, если они являются большими объектами, они идут на кучу больших объектов (LOH), которая иногда называется _поколением 3_. Поколение 3 — это физическое поколение, которое логически собирается как часть поколения 2.
    
    Большинство объектов уничтожается при сборке мусора для поколения 0 и не доживает до следующего поколения.
    
    Если приложение пытается создать новый объект при заполнении поколения 0, сборщик мусора выполняет коллекцию, чтобы освободить адресное пространство для объекта. Сборщик мусора начинает проверять объекты в поколении 0, а не все объекты в управляемой куче. Сборка мусора только в поколении 0 зачастую освобождает достаточно памяти для того, чтобы приложение могло и дальше создавать новые объекты.
    
- **Поколение 1**. Это поколение содержит короткие объекты и служит буфером между короткими объектами и долгосрочными объектами.
    
    Когда сборщик мусора выполняет сборку для поколения 0, память уплотняется для достижимых объектов и они продвигаются в поколение 1. Так как объекты, оставшиеся после сборки, обычно склонны к долгой жизни, имеет смысл продвинуть их в поколение более высокого уровня. Сборщику мусора необязательно выполнять повторную проверку объектов поколений 1 и 2 при каждой сборке мусора поколения 0.
    
    Если коллекция поколения 0 не освобождает достаточно памяти для создания нового объекта, сборщик мусора может выполнить коллекцию поколения 1, а затем поколение 2. Объекты в поколении 1, оставшиеся после сборок, продвигаются в поколение 2.
    
- **Поколение 2**. Это поколение содержит долгосрочные объекты. Примером долгоживущих объектов служит объект в серверном приложении, содержащий статические данные, которые существуют в течение длительности процесса.
    
    Объекты поколения 2, которые сохраняют коллекцию, остаются в поколении 2, пока они не будут недоступны в будущей коллекции.
    
    Объекты в куче больших объектов (иногда называемой _поколением 3_) также собираются в поколении 2.


**LOH (Large Object Heap)** — это область памяти в управляемой куче .NET, предназначенная для хранения больших объектов.

### Что такое LOH?
- Большие объекты — это объекты размером свыше примерно 85 KB.
- Всякий раз, когда приложение создает такой объект, он выделяется в отдельной области памяти, называемой Large Object Heap.
- Например, большие массивы данных, изображения или большие строки обычно попадают в LOH.
### Почему существует LOH?

- Для оптимизации работы сборщика мусора: большие объекты выделяются отдельно, чтобы ускорить управление памятью.
- Разделение помогает избегать фрагментации основного пространства SOH и управлять крупными объектами отдельно.

### Чем LOH отличается от SOH?

|Характеристика|SOH (Small Object Heap)|LOH (Large Object Heap)|
|---|---|---|
|Объекты|Меньше 85 KB|Более 85 KB|
|Назначение|Для небольших объектов|Для больших объектов|
|Управление|Часто собирается вместе|Отдельное управление|

**Корни GC (GC Roots)**
1. Статические поля: Переменные в статических классах или статические члены экземпляров.
2. Локальные переменные: Переменные в текущих выполняющихся методах.
3. Регистры CPU: Значения, хранящиеся в регистрах процессора.
4. GCHandles: Специальные структуры для управления ссылками на объекты (например, для взаимодействия с нативным кодом).
5. Очередь финализации: Объекты, ожидающие вызова деструктора (~ClassName()).

В стеке находятся ссылки на объекты, а сами объекты хранятся в куче. ValueTypes (int, double) не имеют ссылок и сами хранятся в стеке.

**Алгоритм отслеживания ссылок**

Всем объектам в куче задается значение 0 биту в поле индекса блока синхронизации. Затем проверяются все активные корни и объекты, на которые они
ссылаются. Если корень содержит null, он игнорируется, иначе устанавливается бит 1 — это и есть признак маркировки объекта. 
После маркировки объекта CLR проверяет все корни в этом объекте и маркирует объекты, на которые они ссылаются. Встретив уже маркированный объект, сборщик мусора останавливается, чтобы избежать возникновения бесконечного цикла в случае циклических ссылок.

**Коллекции**

**По умолчанию:** Все стандартные коллекции (List, Dictionary, HashSet, Queue, Stack) хранят сильные ссылки на свои элементы.
**Сценарий утечки:** Вы добавили объект в долгоживущую коллекцию, но забыли его удалить из коллекции после того, как он стал не нужен.
**Результат:** Объект остаётся в памяти, потому что коллекция на него ссылается, даже если других ссылок нет.

**Делегаты**

Объект, который инкапсулирует вызов метода.
**Содержит:**
target (или Target): Ссылка на экземпляр объекта, если метод не статический. ЭТО СИЛЬНАЯ ССЫЛКА!
methodptr (или Method): Ссылка на сам метод.
**Итог:** Пока делегат жив, он может удерживать в памяти весь объект, которому принадлежит вызываемый метод.


**Слабые ссылки (Weak References)**
Позволяет ссылаться на объект, но не мешает GC его собрать.
**Как работает:** Если на объект есть только слабые ссылки, GC его соберет. После этого WeakReference вернёт null.
**Применение:** Кэширование, наблюдение за объектами без их удержания.
**Осторожно:** Требует проверки IsAlive или TryGetTarget.
